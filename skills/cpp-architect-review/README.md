# Google C++ 软件架构师审查 Skill

以Google C++软件架构师的标准审查开发方案，提供专业的架构评审和代码质量改进建议。

## 📁 文件结构

```
cpp-architect-review/
├── README.md           # 本文件 - Skill说明
├── SKILL.md           # 主文件 - 审查流程和清单
├── STANDARDS.md       # 核心规范 - 离线可用的编码标准（⭐推荐）
├── REFERENCE.md       # 参考手册 - 详细检查清单和示例
└── EXAMPLES.md        # 实战案例 - 5个完整审查示例
```

## 🎯 功能特点

### 全面的审查维度

1. **架构设计审查** - 模块职责、接口设计、依赖关系、可扩展性
2. **代码质量审查** - 命名规范、函数设计、错误处理、注释规范
3. **性能优化审查** - 内存管理、拷贝优化、资源使用
4. **安全性审查** - 输入验证、内存安全、线程安全
5. **现代C++特性** - 智能指针、optional、variant、移动语义
6. **项目规范检查** - WPS构建系统、文件编码、命名前缀

### 离线可用

✅ **STANDARDS.md** 包含了最常用的审查要点，无需网络即可使用：
- Google C++ Style Guide 核心内容
- C++ Core Guidelines 核心内容  
- 快速检查清单
- 完整的代码示例

### 标准化输出

生成结构化的审查报告：
- 🔴 严重问题 - 必须修复
- 🟡 重要建议 - 建议改进
- 🟢 优化建议 - 可选优化
- ✅ 优点总结
- 📊 评分和后续行动

## 🎓 审查原则

### 1. 基于标准
- Google C++ Style Guide
- C++ Core Guidelines  
- 项目特定规范

### 2. 平衡多个目标
- 正确性 > 安全性 > 可维护性 > 性能 > 简洁性
- 避免过度设计（YAGNI）
- 在可读性和性能间平衡

### 3. 建设性反馈
- 不仅指出问题，更要说明原因
- 提供具体的解决方案
- 肯定好的设计
- 使用专业、尊重的语气

### 4. 关注关键问题
- 优先关注架构层面
- 重点关注正确性和安全性
- 避免纠结风格细节

## 💡 使用技巧

### 快速审查

如果时间紧迫，优先检查：
1. 文件编码（UTF-8 BOM）
2. 资源管理（智能指针、RAII）
3. 错误处理（optional、异常）
4. 线程安全（如果是多线程代码）
5. 明显的性能问题（不必要的拷贝）

### 深度审查

完整审查时，按顺序检查：
1. 架构设计（最重要）
2. 代码质量
3. 安全性
4. 性能优化
5. 规范遵循

### 参考优先级

1. **STANDARDS.md** - 快速查找规范要点（离线）
2. **REFERENCE.md** - 查看详细示例
3. **EXAMPLES.md** - 参考类似案例
4. 在线文档 - 查找完整规范（需要网络）

```markdown
# 架构审查报告: CSS解析器

## 概述
整体设计合理，但存在一些需要改进的问题。

## 严重问题 🔴

### 问题1: 文件编码不是UTF-8 BOM
**位置**: src/core/css/css_parser.cpp
**修复**: 转换为UTF-8 BOM编码

### 问题2: 存在循环依赖
**位置**: css_parser.h ↔ style_sheet.h
**修复**: 使用前向声明

## 重要建议 🟡

### 建议1: 使用string_view避免拷贝
**改进代码**:
\`\`\`cpp
Expected<StyleSheet, ParseError> Parse(std::string_view css);
\`\`\`

## 优化建议 🟢

### 优化1: 容器预留空间
预期提升10-20%性能

## 优点总结 ✅
1. 接口设计清晰
2. 使用智能指针管理资源
3. 单元测试覆盖率高

## 总体评分
- 架构设计: 8/10
- 代码质量: 7/10
- 性能优化: 6/10
- 安全性: 9/10
- 规范遵循: 7/10

## 后续行动
- [ ] 修复文件编码
- [ ] 打破循环依赖
- [ ] 使用string_view优化
```
