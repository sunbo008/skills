# Google C++ 软件架构师审查

以Google C++软件架构师的标准审查开发方案,提供专业的架构评审和代码质量改进建议。

## 审查流程

### 第一步:理解审查目标

首先明确审查的内容:
- 设计文档/技术方案
- 代码实现
- 架构变更
- 性能优化方案

### 第二步:执行多维度审查

按照以下清单进行系统性审查:

#### 1. 架构设计审查

**模块职责检查**:
- [ ] 每个模块是否有单一、明确的职责?
- [ ] 模块边界是否清晰?
- [ ] 是否遵循SOLID原则?

**接口设计检查**:
- [ ] 接口是否简洁、易用?
- [ ] 是否有合理的抽象层次?
- [ ] 接口是否稳定,易于扩展?
- [ ] 是否使用了前向声明减少编译依赖?

**依赖关系检查**:
- [ ] 依赖方向是否正确(高层依赖抽象,不依赖细节)?
- [ ] 是否存在循环依赖?
- [ ] 第三方库使用是否合理?
- [ ] 是否正确使用WPS构建系统的第三方库?

**可扩展性检查**:
- [ ] 设计是否支持未来扩展?
- [ ] 是否使用了合适的设计模式?
- [ ] 是否过度设计(YAGNI原则)?

#### 2. 代码质量审查

**命名规范检查**:
- [ ] 类名是否使用PascalCase?
- [ ] 函数名是否使用PascalCase(Google风格)?
- [ ] 变量名是否使用snake_case或camelCase?
- [ ] 成员变量是否有m_前缀或_后缀?
- [ ] 常量是否使用k前缀+PascalCase?
- [ ] 宏是否使用全大写+下划线?
- [ ] 项目特定前缀是否正确(C/I/E/KSPAICE)?

**文件规范检查**:
- [ ] 所有源文件是否使用UTF-8 BOM编码?
- [ ] 头文件是否有正确的保护(#pragma once或include guard)?
- [ ] include顺序是否正确(对应头文件、C系统、C++标准、第三方、项目)?

**函数设计检查**:
- [ ] 函数是否足够短小(理想20-40行,最大80行)?
- [ ] 函数是否只做一件事?
- [ ] 参数传递是否合理(小类型传值,大类型const引用)?
- [ ] 是否优先使用返回值而非输出参数?
- [ ] 是否使用了std::optional/Expected处理可能失败的操作?

**代码重复检查**:
- [ ] 是否有重复代码需要提取?
- [ ] 是否遵循DRY原则?

**错误处理检查**:
- [ ] 错误处理是否完善?
- [ ] 是否使用了合适的错误处理策略(optional/Expected/异常/错误码)?
- [ ] 边界条件是否处理?
- [ ] 断言使用是否合理?

**注释规范检查**:
- [ ] 是否有文件头注释?
- [ ] 公共类和接口是否有Doxygen注释?
- [ ] 复杂逻辑是否有必要的解释?
- [ ] 是否避免了冗余注释?

#### 3. 性能优化审查

**内存管理检查**:
- [ ] 是否遵循RAII原则?
- [ ] 是否使用智能指针管理资源?
- [ ] 是否避免了不必要的动态分配?
- [ ] 容器是否预留了足够空间(reserve)?

**拷贝优化检查**:
- [ ] 是否使用std::string_view避免字符串拷贝?
- [ ] 是否使用std::span处理数组和缓冲区?
- [ ] 是否积极使用移动语义?
- [ ] 是否使用emplace系列方法就地构造?
- [ ] 参数传递是否避免了不必要的拷贝?

**性能关键路径检查**:
- [ ] 是否有性能瓶颈?
- [ ] 算法复杂度是否合理?
- [ ] 是否有不必要的临时对象?
- [ ] 缓存是否有效利用?

**资源使用检查**:
- [ ] 是否有内存泄漏风险?
- [ ] 文件句柄/网络连接是否正确关闭?
- [ ] 是否有资源竞争?

#### 4. 安全性审查

**输入验证检查**:
- [ ] 所有外部输入是否验证?
- [ ] 是否检查了数组边界?
- [ ] 是否防范了整数溢出?

**内存安全检查**:
- [ ] 是否有缓冲区溢出风险?
- [ ] 是否有use-after-free风险?
- [ ] 是否有空指针解引用风险?
- [ ] 是否正确使用了const保护数据?

**线程安全检查**:
- [ ] 共享数据是否有适当的同步?
- [ ] 是否有死锁风险?
- [ ] 是否正确使用了std::atomic?
- [ ] 是否有数据竞争?

#### 5. 现代C++特性检查

**C++11/14/17/20特性使用**:
- [ ] 是否使用智能指针替代裸指针?
- [ ] 是否使用auto简化类型声明?
- [ ] 是否使用范围for循环?
- [ ] 是否使用nullptr替代NULL?
- [ ] 是否使用override和final?
- [ ] 是否使用[[nodiscard]]防止忽略返回值?
- [ ] 是否使用constexpr进行编译期计算?
- [ ] 是否使用结构化绑定(C++17)?
- [ ] 是否使用std::optional/std::variant(C++17)?

**避免的做法检查**:
- [ ] 是否避免了new/delete?
- [ ] 是否避免了C风格数组?
- [ ] 是否避免了C风格类型转换?
- [ ] 是否避免了在头文件中using namespace std?
- [ ] 是否避免了魔法数字?

#### 6. 项目特定规范检查

**WPS构建系统检查**:
- [ ] 是否正确使用wps_package宏?
- [ ] 是否正确使用wps_use_packages链接第三方库?
- [ ] 是否使用了平台条件语法(WIN/DARWIN/LINUX)?
- [ ] 是否优先使用WPS提供的第三方库?
- [ ] Header-only库是否正确引用?

**目录结构检查**:
- [ ] 文件是否放在正确的目录?
- [ ] 头文件是否在include/目录?
- [ ] 实现是否在src/目录?
- [ ] 测试是否在tests/目录?

### 第四步:提供修复方案

对于严重问题和重要建议,提供具体的修复代码:

1. **直接修复**: 如果问题明确且修复简单,直接提供修复后的完整代码
2. **重构建议**: 如果需要重构,提供重构步骤和关键代码片段
3. **替代方案**: 如果有多种解决方案,列出各方案的优缺点

## 审查原则

### 1. 基于标准和最佳实践
- Google C++ Style Guide
- C++ Core Guidelines
- 项目特定规范(WPS构建系统、KSPAICE命名规范)

### 2. 平衡多个目标
- 正确性 > 安全性 > 可维护性 > 性能 > 简洁性
- 避免过度设计,遵循YAGNI原则
- 在可读性和性能之间找到平衡

### 3. 提供建设性反馈
- 不仅指出问题,更要说明原因和解决方案
- 肯定好的设计和实现
- 使用专业、尊重的语气

### 4. 关注关键问题
- 优先关注架构层面的问题
- 重点关注影响正确性和安全性的问题
- 避免纠结于风格细节

## 常见问题模式

### 架构问题
- 模块职责不清,上帝类
- 紧耦合,难以测试
- 循环依赖
- 过度设计或设计不足

### 代码质量问题
- 命名不规范或不清晰
- 函数过长,做太多事
- 重复代码
- 错误处理不完善

### 性能问题
- 不必要的拷贝
- 缺少移动语义
- 容器未预留空间
- 临时对象过多

### 安全问题
- 缺少输入验证
- 内存管理不当
- 线程安全问题
- 资源泄漏

### 规范问题
- 文件编码不是UTF-8 BOM
- 命名不符合规范
- 未使用现代C++特性
- 注释不规范

## 注意事项

1. **保持客观**: 基于标准和最佳实践,不掺杂个人偏好
2. **提供依据**: 每个问题和建议都要说明原因和参考标准
3. **考虑上下文**: 理解项目约束和特殊需求
4. **优先级明确**: 区分严重问题、重要建议和优化建议
5. **可执行性**: 提供的修复方案要具体可行
